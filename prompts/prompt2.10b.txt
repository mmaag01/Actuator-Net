# Context

Test 5b established that torEst flips sign at the failure sample (+103 → -107 Nm) rather than decaying gracefully, meaning torEst during uncommanded phases actively misinforms the model rather than just being uninformative. The state-carrier features need to be magnitude- or history-based, not raw-polarity based.

This supersedes Prompt 9 from the previous turn. Run after Prompt 10a confirms that some non-trivial fraction of failures can't be fixed by window length alone.

# Task

Part A — hardware signal audit (unchanged from original Prompt 9). Read hej90datasheet.pdf and EPOS4_ANYbotics_ANYdrive_Firmware_Specification_Additional_140h.pdf. Confirm whether posAct is motor-side or load-side, whether a second position signal exists in the register map, and the spring stiffness k_spring.

Additionally for Part A: verify the hypothesis that torEst = k_t · i_q. In the Test 5b failure window, compute the ratio torEst / i over the 30 samples. If constant, that's the k_t and confirms torEst is iq-derived. Report the inferred k_t and compare to datasheet-rated k_t if available.

Write findings to `experiments/outputs/hej90_signal_audit.md`.

Part B — engineered features. Construct FIVE candidate state-carrier features, computed in dataset.py before windowing, with enforced causal offset (use `assert` that no feature at time t uses torAct at time t-49 or later):

1. **`torAct_ema_200ms_lag50`**: EMA of torAct with 200 ms time constant, evaluated at t-50 samples. Standard candidate.

2. **`torAct_max_abs_500ms_lag50`**: max of |torAct| over [t-550, t-50] samples. Magnitude-based, polarity-invariant.

3. **`torEst_max_abs_500ms_lag50`**: max of |torEst| over [t-550, t-50]. Crucially, this is available at inference time WITHOUT needing a lagged torAct feed — torEst is in the live feature set. The hypothesis: during a failure sample torEst has already flipped sign and is misleading, but its MAX over the last half second still reflects the loading peak.

4. **`torEst_sign_flip_count_200ms`**: count of sign changes in torEst over the last 200 ms. Should be ~0 during steady loading, spike during discharge/ringing events.

5. **`torAct_minus_torEst_ema_200ms_lag50`**: EMA of the residual between torAct and torEst over the past (with the 50-sample offset). This encodes how much the two torque signals have recently been disagreeing — which is precisely the signal of spring-stored energy.

All five must pass an automated causality check: a test that the feature value at time t depends only on signals at time ≤ t-50.

Part C — retrain and evaluate. Start with SEQ_LEN=30 baseline (isolates the feature effect from the window effect). Retrain MLP and GRU with each feature added independently, then all five together. Six configs × 2 models = 12 runs.

# Interpretation

- "Magnitude-based feature is decisive" if max_torAct or max_torEst is the single biggest winner — thesis narrative is about polarity-invariant state encoding
- "Residual feature is decisive" if residual_ema wins — thesis narrative is that the torAct/torEst disagreement IS the state signal
- "Combination wins cleanly" if all-five outperforms any single feature by >20% — thesis is about feature engineering for series-elastic actuators  
- "No recovery possible" if nothing reduces idx=38180 error by >50% — thesis finding is that EPOS4 telemetry alone is fundamentally insufficient for this failure class
