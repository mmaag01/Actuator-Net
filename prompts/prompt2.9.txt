# Context

If Prompt 7 verdicts "torEst follows command" OR Prompt 8's SEQ_LEN=1000 still fails idx=38180, then the problem isn't lookback length — it's that the available features don't observe spring state during uncommanded phases. The HEJ-90 is a series-elastic actuator: torAct is measured from spring deflection; torEst is derived from motor current (iq via EPOS4_ANYbotics_ANYdrive_Firmware_Specification_Additional_140h.pdf).

When the motor is commanded off, iq goes to zero, so torEst goes to zero, even if the spring is still deflected. The spring state is physically carried by the DIFFERENCE between motor-side and load-side positions, but we need to confirm what position signals are actually in the dataset.

# Task

1. Read the HEJ-90 datasheet (project file: hej90datasheet.pdf) and the ANYbotics EPOS4 addendum (EPOS4_ANYbotics_ANYdrive_Firmware_Specification_Additional_140h.pdf). Confirm:

   - What is the spring stiffness k_spring in Nm/rad?

   Write findings to `experiments/outputs/hej90_signal_audit.md`.

Part B — engineered features (1 day):

Construct three candidate "state-carrier" features from existing data, each intended to encode recent torque momentum without directly leaking the target:

1. **`torAct_ema_200ms`**: exponentially-weighted moving average of torAct with time constant 200 ms, evaluated at t-50 samples (50 ms before the prediction target). The 50-sample offset is the critical design choice — it ensures the feature isn't just a lagged copy of the target. Justify in a comment that 50 ms is ~2.5× the typical ringing decay time constant, so this feature sees the "recent past" without being a cheat.

2. **`torAct_max_abs_500ms`**: max of |torAct| over [t-550, t-50] samples. Tells the model "was there recent loading?"

3. **`posAct_minus_posDes_integrated_200ms`**: integral of the position tracking error over the prior 200 ms. If posAct is motor-side and posDes encodes the commanded position, this is a rough proxy for accumulated mechanical strain. Only include if Part A confirms posAct is motor-side.

All three computed in `dataset.py` as new columns before windowing. Reuses the existing SG-smoothing code path pattern.

# Interpretation

- "State carrier identified" if the winning config reduces idx=38180 error by >70% AND overall RMSE by >15%
- "Partial state recovery" if idx=38180 improves but overall RMSE barely moves (means the state feature helps the specific failure mode but introduces noise elsewhere)
- "No state recovery possible" if nothing works — this is itself a thesis finding: EPOS4 telemetry alone is insufficient for series-elastic-actuator modeling during uncommanded phases.

# Requirements
